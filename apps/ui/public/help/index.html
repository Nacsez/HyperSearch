Overview

What HyperSearch Does

HyperSearch is an API-first local research service. It sends web queries to SearXNG, optionally fetches result pages, extracts text, and can ask a local model provider such as LM Studio to synthesize an answer with citations. The web UI is a first-party operator console on top of the same backend endpoints that headless clients use.

Search mode

Returns ranked search results quickly. This is the fastest path and should be your default when you want links, snippets, caching behavior, and basic summaries.

Research mode

Pulls top results, fetches page content, extracts readable text, and asks the active provider to synthesize an answer with citations and a debug trace.

Observed Behavior

What The Recent Logs Show

  • Basic search requests are healthy and usually completed in about 1.0 to 1.1 seconds for simple runs.
  • Broader search runs with more payload or more engines were closer to 3.3 to 3.8 seconds.
  • Provider health tests against LM Studio completed successfully in about 3.8 to 5.2 seconds.
  • Research runs varied a lot: one completed in about 6.3 seconds, while a deeper run took about 49.2 seconds.
  • Some target sites returned 403 Forbidden during fetch and extract. That is expected on parts of the public web and should not be treated as a HyperSearch failure by itself.
  • Trafilatura occasionally logged discarding data: None. That is a recoverable extraction edge case, not a crash.
  • Caddy access logs showed clean end-to-end proxying with request IDs and trace IDs preserved on API responses.
Operational takeaway:

Search latency is currently good. Research latency depends heavily on the target pages and fetchability. Your test plan should therefore cover both fast, cooperative sources and slower or blocked sources.

Reference

Field Guide

Field Used By Meaning Operator Notes
Query Search, Research The search phrase sent to SearXNG. Use focused, factual queries first. Broader wording increases result variety but can increase noise.
Engines Search, Research Comma-separated engine list such as google,duckduckgo. Leave blank to use backend defaults. Restrict this when comparing engine behavior.
Categories Search, Research Comma-separated SearXNG categories such as general,news. Useful for isolating news-like results or narrowing broad searches.
Language Search, Research Language hint, for example en-US. Use when you need locale-sensitive ranking or local-language material.
Date Range Search, Research Freshness filter such as past day, week, month, or year when the selected search engine supports it. Useful for current events. Leave blank for general web discovery.
Results to Collect Search, Research Total number of search results HyperSearch should collect. The UI calculates backend page settings automatically. Increase this when you need deeper recall; larger values are slower and produce more local cache entries.
Research Sources Research How many collected search hits to fetch, summarize, and synthesize. Start around 3 to 5. The default 1.0 bound is 250; large values can make research much slower.
Safe Search Search, Research SearXNG safety mode: off, moderate, or strict. Leave at moderate for most use.
Cache Policy Search, Research Controls cache usage for search, fetch, extract, and synthesis paths. use is normal. bypass is best for fresh verification. refresh forces recomputation and updates cache.
Dedupe Search Removes duplicate or near-duplicate results. Usually leave on unless you are analyzing engine overlap.
Fetch pages Search Fetches linked pages after search. Use this when snippets are not enough and you want body content attached to results.
Extract text Search Runs extraction on fetched pages. Best paired with Fetch pages. Useful when snippets are noisy or truncated.
Search summary Search Requests a compact search-level summary. Good for quick scans without invoking full research mode.
Provider Override Research Forces research synthesis through a specific configured provider. Leave blank to use the default provider. Useful for A/B comparison across providers later.
Timeout (ms) Search, Research Request timeout budget. Raise this for heavy research runs. A short timeout is a good stress test input.
Headless-only options:

The API also exposes options like streaming and include_debug_trace. These are useful when integrating HyperSearch into scripts, agents, or other services.

Workflow

Presets And Profiles

The current UI uses a preset system. If you think of these as search profiles, that is the right mental model. A preset stores the current request shape so you can re-run common tasks without re-entering all fields.

  • Set the fields you want in the search form.
  • Click Save Preset.
  • Give the preset a descriptive name such as Recent news sweep or Wikipedia fact check.
  • Use the saved preset chips at the bottom of the search form to load it later.
  • After loading a preset, you can still change individual fields before running the next request.
Recommended naming pattern:

Use names that encode both intent and shape, for example News month fast, Deep research 5 pages, or English general bypass cache.

Export

Session XML Export

Session XML export is meant for durable personal archives, notebook ingestion, and downstream automation. Browser sessions download the XML through the browser. Embedded desktop sessions hand the XML to the desktop shell, which writes the file into the export folder configured in desktop settings.

File naming

Exported files use the first 48 characters of the active query, sanitized for Windows filenames, followed by a timestamp such as 20260426_153015.xml.

Included fields

Open Search Settings, then XML export contents, to choose whether the export contains metadata, the question, search results, fetched full text, summaries, research answer, research sources, activity log, and provider trace.

Privacy note:

XML exports can include scraped page text, local provider/model metadata, and detailed activity logs. Turn off fields you do not need before sharing an export outside your own machine.

Exercises

UI Walkthroughs

Exercise 1: Basic search in the UI

  1. Open the main app at http://127.0.0.1:8090/.
  2. Enter history of cincinnati ohio as the query.
  3. Set Results to Collect to 10.
  4. Leave cache policy at use for the first run.
  5. Click Run Search.
  6. Verify that the workspace shows result count, request ID, result cards, and cache state.
  7. Open Verbose log and confirm the raw payload for the request exists.

Exercise 2: Research synthesis in the UI

  1. Set query to origin of ronald mcdonald.
  2. Set Research Sources = 3 for a moderate run.
  3. Optionally open advanced controls and set a larger timeout such as 60000.
  4. Click Research Synthesis.
  5. Verify that the answer panel fills in above the collected source list.
  6. Check that an answer appears with numbered citations and a trace summary.
  7. Open the debug trace details and verify the request completed even if some pages were blocked.

API

Headless Usage

Headless use goes through the same backend as the UI. Local requests on localhost do not need a token. LAN access must be enabled in the desktop launcher and paired clients should supply X-HyperSearch-Token.

PowerShell search example

.\scripts\Invoke-HyperSearch.cmd `
  -Action search `
  -Query "history of cincinnati ohio" `
  -Results 10 `
  -Summarize
{
  "query": "history of cincinnati ohio",
  "page": 1,
  "results_per_page": 5,
  "max_pages": 1,
  "safe_search": 1,
  "dedupe": true,
  "fetch_pages": false,
  "extract_text": false,
  "summarize": true,
  "streaming": false,
  "timeout_ms": 15000,
  "cache_policy": "use"
}

$body = @'
{
  "query": "history of cincinnati ohio",
  "page": 1,
  "results_per_page": 5,
  "max_pages": 1,
  "safe_search": 1,
  "dedupe": true,
  "fetch_pages": false,
  "extract_text": false,
  "summarize": true,
  "streaming": false,
  "timeout_ms": 15000,
  "cache_policy": "use"
}
'@

Invoke-RestMethod `
  -Uri http://127.0.0.1:8090/v1/search `
  -Method Post `
  -ContentType "application/json" `
  -Body $body

PowerShell research example

.\scripts\Invoke-HyperSearch.cmd `
  -Action research `
  -Query "origin of ronald mcdonald" `
  -Results 25 `
  -ResearchSources 5 `
  -TimeoutMs 60000
$body = @'
{
  "query": "origin of ronald mcdonald",
  "page": 1,
  "results_per_page": 10,
  "max_pages": 1,
  "top_n": 3,
  "timeout_ms": 60000,
  "cache_policy": "use",
  "provider": null,
  "streaming": false,
  "include_debug_trace": true
}
'@

Invoke-RestMethod `
  -Uri http://127.0.0.1:8090/v1/research `
  -Method Post `
  -ContentType "application/json" `
  -Body $body

Provider and history automation

.\scripts\Invoke-HyperSearch.cmd -Action providers
.\scripts\Invoke-HyperSearch.cmd -Action provider-models -ProviderName lmstudio
.\scripts\Invoke-HyperSearch.cmd `
  -Action set-provider `
  -ProviderName lmstudio `
  -DisplayName "LM Studio" `
  -BaseProviderUrl "http://host.docker.internal:1234" `

.\scripts\Invoke-HyperSearch.cmd -Action history-export

Deploy

Installation And Deployment

Release candidate install

  1. Use the full installer media for private beta tests whenever possible.
  2. Run HyperSearch_1.0.0_x64-setup.exe from the media folder.
  3. If payload\images is present, setup loads Docker images locally with docker load.
  4. If no image archive is present, setup uses online mode and pulls prebuilt images.
  5. Review the setup result screen before closing the installer.
  6. Launch HyperSearch Desktop and press Start.
  7. Keep LAN mode disabled unless another local-network device needs access.

Local development

  1. Create a Python virtual environment and install the API package from apps/api.
  2. Install UI dependencies in apps/ui with npm install.
  3. Run the API with Uvicorn on 127.0.0.1:8000.
  4. Run the UI with npm run dev on 127.0.0.1:5173.
  5. For Compose release mode, run docker compose --project-name hypersearch up -d.
  6. For local image builds, add -f docker-compose.yml -f docker-compose.dev.yml up -d --build.

Docker Desktop

  • Setup detects Docker Desktop and can guide installation when missing.
  • Startup requires a clean Docker engine version response.
  • Online media requires registry/DNS/proxy access for image pulls.
  • Full media can start without registry access after images are loaded.

LM Studio and models

  • Research synthesis requires a local OpenAI-compatible endpoint.
  • Docker-managed HyperSearch usually reaches LM Studio at http://host.docker.internal:1234.
  • Low-resource systems may be configured as search-only until a suitable model is selected.
  • Use Operations to discover loaded models, select a model, save the provider, and run a provider test.
Network exposure:

HyperSearch is designed to bind to localhost by default. LAN mode is opt-in from the desktop launcher and paired browsers must send the generated pairing token. Do not expose HyperSearch to the public internet.

Diagnostics:

Installer logs, setup summaries, desktop logs, and full command logs are written under %LOCALAPPDATA%\HyperSearch\logs. Use desktop Settings > Export Diagnostics before filing a beta issue report.

Validation

Thorough Testing Plan

Phase 1: Stack health

  1. Open the UI and confirm the shell loads without layout breakage.
  2. Open the help page from the Help button and verify it loads in a browser tab or a desktop session titled Help.
  3. Open Operations and run a provider test.
  4. Verify provider status is healthy and response time is within a reasonable few-second range.
  5. Confirm the verbose log records admin events.

Phase 2: Core search behavior

  1. Run a simple factual search with cache policy use.
  2. Run the same search again and verify a more cache-heavy result path.
  3. Run again with cache policy bypass and confirm it still succeeds.
  4. Toggle Dedupe off and compare the number and similarity of results.
  5. Change Results to Collect from 5 to 20 and verify payload size and UI handling remain solid.
  6. Use advanced controls to restrict engines and categories, then verify the returned result mix changes.

Phase 3: Fetch and extract behavior

  1. Run search with Fetch pages enabled on a cooperative source like Wikipedia.
  2. Run search with Fetch pages and Extract text enabled.
  3. Confirm some result cards show fetched and extracted markers.
  4. Repeat with a more hostile query that is likely to include blocked domains and confirm the request still completes.

Phase 4: Research behavior

  1. Run a short research request with Research Sources = 2 or 3.
  2. Confirm answer text, citation cards, and trace summary all appear.
  3. Run the same research again with refresh cache policy.
  4. Raise Research Sources and timeout to simulate a slower research path.
  5. Verify long-running research still completes and the UI remains usable while waiting.

Phase 5: Headless API checks

  1. Run the search PowerShell example from this page and confirm a request_id is returned.
  2. Run the research PowerShell example and verify citations plus trace fields are present.
  3. Capture one response and verify request IDs in the payload line up with the API logs.
  4. If you plan LAN access, repeat the same calls with X-HyperSearch-Token set.

Phase 6: Regression passes

  1. Save a preset, reload it, and verify fields rehydrate correctly.
  2. Toggle light and dark mode and confirm the drawer and workspace remain readable.
  3. Resize the browser to a narrower width and verify controls and drawers stay usable.
  4. Use the verbose log after several actions and confirm event selection still maps to the right payload.

Quick Start

Most common operator loop

  1. Confirm provider health.
  2. Run search first.
  3. Inspect results.
  4. Escalate to research only when you need synthesis.
  5. Use verbose log when you need raw payloads.

Known Behaviors

Do not misread these as failures

  • Some pages on the web will block automated fetches with 403 responses.
  • Provider tests are usually slower than search because they hit the local model backend.
  • Research duration can jump sharply when top N is high or sources are heavy.
